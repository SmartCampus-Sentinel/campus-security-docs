## 3. 文档管理规范
### 3.1 文档命名规范
- 命名格式：`文档类型-文档名称.md`（如：设计文档-数据库设计文档.md）
- 迭代相关文档：`文档类型-文档名称-迭代X.md`（如：测试报告-迭代2测试报告.md）
- 禁止使用特殊字符，文件名统一使用中文

### 3.2 文档更新规范
- 需求变更、设计调整后，需在1个工作日内更新对应文档
- 文档更新后，需在提交说明中注明更新内容和原因
- 重要文档（如需求规格说明书、部署手册）更新后，需通知团队成员

### 3.3 文档查阅方式
- 本地查阅：克隆仓库后，按目录结构查找对应文档
- 在线查阅：直接访问GitHub仓库，通过目录导航查看

## 4. 仓库使用说明
### 4.1 克隆仓库
`git clone https://github.com/【组织名称】/campus-security-docs.git`

### 4.2 文档贡献流程
1. 从dev分支创建文档编辑分支，命名格式：docs/文档类型-文档名称（如：docs/设计文档-接口规范）
2. 在分支中编写/修改文档
3. 提交代码（提交规范：`docs(文档类型): 文档更新说明`，示例：`docs(设计文档): 补充数据库表关系说明`）
4. 提交PR到dev分支，指定文档维护人员审核
5. 审核通过后合并至dev分支，定期合并dev分支到master分支

## 5. 分支管理规范
- master：稳定版本分支，存放最终定稿的文档，禁止直接提交
- dev：开发主分支，存放最新的文档版本，供团队查阅
- docs/xxx：文档编辑分支，从dev创建，完成后合并至dev

## 6. 提交规范
提交信息格式：`docs(文档类型): description`
- 文档类型：需求文档、设计文档、开发规范、测试文档、部署文档等
- description：简要描述文档更新内容（中文）
- 示例：`docs(部署文档): 补充Docker Compose部署步骤`
- 示例：`docs(测试文档): 新增AI分析服务测试用例`

## 7. 核心文档说明

- 说明：规定团队的仓库管理、代码提交、PR流程、迭代周期等协作规则
### 7.4 部署手册
- 位置：部署文档/生产环境部署手册.md
- 说明：详细描述生产环境的服务器配置、部署步骤、启动命令、故障处理等

## 8. 常见问题排查
- 问题1：克隆仓库后无法查看文档格式
  解决方案：确保本地编辑器支持Markdown格式（如VS Code+Markdown All in One插件），或直接在GitHub在线查看
- 问题2：找不到最新的需求变更内容
  解决方案：查看需求文档/需求变更记录.md，或切换到dev分支查看最新文档版本
- 问题3：文档内容与代码实现不一致
  解决方案：在GitHub Issues中创建问题，标注文档名称和不一致位置，通知文档维护人员更新
